# Repository Guidelines

## Project Structure & Module Organization
- `lib/` contains the plugin implementation, organized under `Membrane.MPEG.TS` (e.g., `lib/membrane/mpeg/ts/demuxer.ex`).
- `test/` holds ExUnit tests (`test/membrane/mpeg/ts/*_test.exs`) plus helpers in `test/support/`.
- `test/data/` includes binary fixtures (`*.ts`, `*.aac`, `*.h264`) used by tests.
- `config/` has runtime configuration (`config/config.exs`).
- `doc/` is generated by `mix docs`.

## Build, Test, and Development Commands
- `mix deps.get` fetches dependencies.
- `mix compile` builds the project.
- `mix test` runs the full ExUnit suite.
- `mix format` formats Elixir sources based on `.formatter.exs`.
- `mix docs` generates API docs into `doc/`.

## Coding Style & Naming Conventions
- Use standard Elixir formatting (2-space indentation) and run `mix format` before pushing.
- Module names follow `Membrane.MPEG.TS.*` and map to file paths under `lib/membrane/mpeg/ts/`.
- Test files end with `_test.exs` and mirror module paths (e.g., `test/membrane/mpeg/ts/demuxer_test.exs`).

## Testing Guidelines
- Tests use ExUnit (`test/test_helper.exs`), with fixtures in `test/data/`.
- Prefer focused unit tests for demuxer/muxer behaviors and add fixtures when introducing new stream cases.
- Run `mix test` locally; add new tests alongside related modules.

## Commit & Pull Request Guidelines
- Commit messages are short, imperative, and usually capitalized (e.g., “Improve testing”, “Update mpeg_ts”, “Bump version”).
- PRs should include a concise description, the motivation/impact, and the tests run.
- Link related issues and update docs/fixtures when behavior changes.

## Environment & Tooling
- Tooling is managed with `mise.toml` (Elixir 1.19.4 / OTP 28.2). Ensure your local toolchain matches before running builds.
